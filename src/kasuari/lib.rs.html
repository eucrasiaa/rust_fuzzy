1//! This crate contains an implementation of the Cassowary constraint solving algorithm, based upon
2//! the work by G.J. Badros et al. in 2001. This algorithm is designed primarily for use
3//! constraining elements in user interfaces. Constraints are linear combinations of the problem
4//! variables. The notable features of Cassowary that make it ideal for user interfaces are that it
5//! is incremental (i.e. you can add and remove constraints at runtime and it will perform the
6//! minimum work to update the result) and that the constraints can be violated if necessary, with
7//! the order in which they are violated specified by setting a "strength" for each constraint. This
8//! allows the solution to gracefully degrade, which is useful for when a user interface needs to
9//! compromise on its constraints in order to still be able to display something.
10//!
11//! ## Constraint syntax
12//!
13//! This crate aims to provide syntax for describing linear constraints as naturally as possible,
14//! within the limitations of Rust's type system. Generally you can write constraints as you would
15//! naturally, however the operator symbol (for greater-than, less-than, equals) is replaced with an
16//! instance of the `WeightedRelation` enum wrapped in "pipe brackets".
17//!
18//! For example, for the constraint `(a + b) * 2 + c >= d + 1` with strength `s`, the code to use is
19//!
20//! ```ignore
21//! (a + b) * 2.0 + c |GE(s)| d + 1.0
22//! ```
23//!
24//! # A simple example
25//!
26//! Imagine a layout consisting of two elements laid out horizontally. For small window widths the
27//! elements should compress to fit, but if there is enough space they should display at their
28//! preferred widths. The first element will align to the left, and the second to the right. For
29//! this example we will ignore vertical layout.
30//!
31//! First we need to include the relevant parts of `cassowary`:
32//!
33//! ```
34//! use kasuari::WeightedRelation::*;
35//! use kasuari::{Solver, Variable};
36//! ```
37//!
38//! And we'll construct some conveniences for pretty printing (which should hopefully be
39//! self-explanatory):
40//!
41//! ```ignore
42//! use hashbrown::HashMap;
43//! let mut names = HashMap::new();
44//! fn print_changes(names: &HashMap<Variable, &'static str>, changes: &[(Variable, f64)]) {
45//!     println!("Changes:");
46//!     for &(ref var, ref val) in changes {
47//!         println!("{}: {}", names[var], val);
48//!     }
49//! }
50//! ```
51//!
52//! Let's define the variables required - the left and right edges of the elements, and the width of
53//! the window.
54//!
55//! ```ignore
56//! let window_width = Variable::new();
57//! names.insert(window_width, "window_width");
58//!
59//! struct Element {
60//!     left: Variable,
61//!     right: Variable
62//! }
63//! let box1 = Element {
64//!     left: Variable::new(),
65//!     right: Variable::new()
66//! };
67//! names.insert(box1.left, "box1.left");
68//! names.insert(box1.right, "box1.right");
69//!
70//! let box2 = Element {
71//!     left: Variable::new(),
72//!     right: Variable::new()
73//! };
74//! names.insert(box2.left, "box2.left");
75//! names.insert(box2.right, "box2.right");
76//! ```
77//!
78//! Now to set up the solver and constraints.
79//!
80//! ```ignore
81//! let mut solver = Solver::new();
82//! solver.add_constraints(&[
83//!     window_width |GE(REQUIRED)| 0.0, // positive window width
84//!     box1.left |EQ(REQUIRED)| 0.0, // left align
85//!     box2.right |EQ(REQUIRED)| window_width, // right align
86//!     box2.left |GE(REQUIRED)| box1.right, // no overlap
87//!     // positive widths
88//!     box1.left |LE(REQUIRED)| box1.right,
89//!     box2.left |LE(REQUIRED)| box2.right,
90//!     // preferred widths:
91//!     box1.right - box1.left |EQ(WEAK)| 50.0,
92//!     box2.right - box2.left |EQ(WEAK)| 100.0
93//! ])?;
94//! # Ok::<(), kasuari::InternalSolverError>(())
95//! ```
96//!
97//! The window width is currently free to take any positive value. Let's constrain it to a
98//! particular value. Since for this example we will repeatedly change the window width, it is most
99//! efficient to use an "edit variable", instead of repeatedly removing and adding constraints (note
100//! that for efficiency reasons we cannot edit a normal constraint that has been added to the
101//! solver).
102//!
103//! ```ignore
104//! solver.add_edit_variable(window_width, STRONG).unwrap();
105//! solver.suggest_value(window_width, 300.0).unwrap();
106//! ```
107//!
108//! This value of 300 is enough to fit both boxes in with room to spare, so let's check that this is
109//! the case. We can fetch a list of changes to the values of variables in the solver. Using the
110//! pretty printer defined earlier we can see what values our variables now hold.
111//!
112//! ```ignore
113//! print_changes(&names, solver.fetch_changes());
114//! ```
115//!
116//! This should print (in a possibly different order):
117//!
118//! ```ignore
119//! Changes:
120//! window_width: 300
121//! box1.right: 50
122//! box2.left: 200
123//! box2.right: 300
124//! ```
125//!
126//! Note that the value of `box1.left` is not mentioned. This is because `solver.fetch_changes` only
127//! lists *changes* to variables, and since each variable starts in the solver with a value of zero,
128//! any values that have not changed from zero will not be reported.
129//!
130//! Now let's try compressing the window so that the boxes can't take up their preferred widths.
131//!
132//! ```ignore
133//! solver.suggest_value(window_width, 75.0);
134//! print_changes(&names, solver.fetch_changes);
135//! ```
136//!
137//! Now the solver can't satisfy all of the constraints. It will pick at least one of the weakest
138//! constraints to violate. In this case it will be one or both of the preferred widths. For
139//! efficiency reasons this is picked nondeterministically, so there are two possible results. This
140//! could be
141//!
142//! ```ignore
143//! Changes:
144//! window_width: 75
145//! box1.right: 0
146//! box2.left: 0
147//! box2.right: 75
148//! ```
149//!
150//! or
151//!
152//! ```ignore
153//! Changes:
154//! window_width: 75
155//! box2.left: 50
156//! box2.right: 75
157//! ```
158//!
159//! Due to the nature of the algorithm, "in-between" solutions, although just as valid, are not
160//! picked.
161//!
162//! In a user interface this is not likely a result we would prefer. The solution is to add another
163//! constraint to control the behaviour when the preferred widths cannot both be satisfied. In this
164//! example we are going to constrain the boxes to try to maintain a ratio between their widths.
165//!
166//! ```
167//! # use kasuari::{ Solver, Variable, Strength };
168//! # use kasuari::WeightedRelation::*;
169//! #
170//! # use hashbrown::HashMap;
171//! # let mut names = HashMap::new();
172//! # fn print_changes(names: &HashMap<Variable, &'static str>, changes: &[(Variable, f64)]) {
173//! #     println!("Changes:");
174//! #     for &(ref var, ref val) in changes {
175//! #         println!("{}: {}", names[var], val);
176//! #     }
177//! # }
178//! #
179//! # let window_width = Variable::new();
180//! # names.insert(window_width, "window_width");
181//! # struct Element {
182//! #    left: Variable,
183//! #    right: Variable
184//! # }
185//! # let box1 = Element {
186//! #     left: Variable::new(),
187//! #     right: Variable::new()
188//! # };
189//! # names.insert(box1.left, "box1.left");
190//! # names.insert(box1.right, "box1.right");
191//! # let box2 = Element {
192//! #     left: Variable::new(),
193//! #     right: Variable::new()
194//! # };
195//! # names.insert(box2.left, "box2.left");
196//! # names.insert(box2.right, "box2.right");
197//! # let mut solver = Solver::new();
198//! # solver.add_constraints([
199//! #     window_width |GE(Strength::REQUIRED)| 0.0, // positive window width
200//! #     box1.left |EQ(Strength::REQUIRED)| 0.0, // left align
201//! #     box2.right |EQ(Strength::REQUIRED)| window_width, // right align
202//! #     box2.left |GE(Strength::REQUIRED)| box1.right, // no overlap
203//! #     // positive widths
204//! #     box1.left |LE(Strength::REQUIRED)| box1.right,
205//! #     box2.left |LE(Strength::REQUIRED)| box2.right,
206//! #     // preferred widths:
207//! #     box1.right - box1.left |EQ(Strength::WEAK)| 50.0,
208//! #     box2.right - box2.left |EQ(Strength::WEAK)| 100.0]).unwrap();
209//! # solver.add_edit_variable(window_width, Strength::STRONG).unwrap();
210//! # solver.suggest_value(window_width, 300.0).unwrap();
211//! # print_changes(&names, solver.fetch_changes());
212//! # solver.suggest_value(window_width, 75.0);
213//! # print_changes(&names, solver.fetch_changes());
214//! solver.add_constraint(
215//!     (box1.right - box1.left) / 50.0 |EQ(Strength::MEDIUM)| (box2.right - box2.left) / 100.0
216//!     ).unwrap();
217//! print_changes(&names, solver.fetch_changes());
218//! ```
219//!
220//! Now the result gives values that maintain the ratio between the sizes of the two boxes:
221//!
222//! ```ignore
223//! Changes:
224//! box1.right: 25
225//! box2.left: 25
226//! ```
227//!
228//! This example may have appeared somewhat contrived, but hopefully it shows the power of the
229//! cassowary algorithm for laying out user interfaces.
230//!
231//! One thing that this example exposes is that this crate is a rather low level library. It does
232//! not have any inherent knowledge of user interfaces, directions or boxes. Thus for use in a user
233//! interface this crate should ideally be wrapped by a higher level API, which is outside the scope
234//! of this crate.
